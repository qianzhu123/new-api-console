# 地址签到设置与今日手动标记设计

## 背景

地址的签到能力和“今日是否已人工完成签到”当前共用账号签到状态。`api.e2ez.com` 又被硬编码为强制不可签到，因此即使后端已经写入“已签到”，公开账号数据和前端分组判断仍会把它覆盖为“不可签到”，造成右上角提示成功而左侧手动签到标记不更新。

状态检测接口还会根据远端 `checkin_enabled` 直接把地址设置改成“可以签到”或“不可签到”，会覆盖用户已经选择的“手动签到”或“不可签到”。

## 目标行为

签到设置和今日手动标记是两个独立维度：

- `可以签到`：允许自动签到和批量签到。检测签到状态可以更新该地址的签到能力。
- `手动签到`：不执行自动签到；显示紧凑的今日手动标记按钮。左侧地址栏显示“手动签到 · 今日已签到/今日未签到”。
- `不可签到`：不执行自动签到；今日签到状态始终显示“不可签到”。详情区仍显示紧凑的今日手动标记按钮，但该标记不改变“不可签到”的状态文案。
- 当前设置不是“可以签到”时，检测签到状态不得修改签到设置。

## 数据与接口设计

后端继续持久化地址下账号的当日签到记录，并新增明确的地址级读取字段 `daily_signin_marked`，供地址详情和左侧手动签到标记使用。该字段读取原始当日记录，不经过“强制不可签到”或“手动禁用”的公开状态归一化。

现有地址手动标记接口调整为只切换今日标记，不再把地址模式强制改成 `manual`。接口响应同时返回更新后的地址信息和 `daily_signin_marked`，前端可以直接更新缓存，不依赖公开账号状态反推。

地址信息 GET 响应也包含 `daily_signin_marked`，确保刷新页面后标记保持一致。

签到状态检测接口读取当前地址模式：

- 当前为 `enabled`：照常检测并根据结果更新为 `enabled` 或 `disabled`。
- 当前为 `manual` 或 `disabled`：可以执行远端能力检测并返回结果，但不写入 `checkin_mode`，也不清除或覆盖今日手动标记。

## 前端设计

前端的手动标记展示改为读取地址信息中的 `daily_signin_marked`，不再通过账号公开签到状态计算。

详情区在 `manual` 和 `disabled` 两种模式下显示同一套紧凑控件：

- 未标记：`标记今日已签到`
- 已标记：`今日已签到`

控件使用小号按钮和收紧的容器间距，尺寸低于当前手动签到卡片。`manual` 模式下，左侧地址栏标记随按钮立即切换；`disabled` 模式下，左侧统计和账号今日签到状态仍显示“不可签到”。

分组是否不可签到只由签到能力决定。`manual` 不能再被描述为“网站不支持签到”；其批量签到按钮可以禁用，但文案应表达“手动签到”，避免与 `disabled` 混淆。

## 错误处理

- 手动标记请求失败时不修改本地缓存，保留原显示并提示错误。
- 检测失败时保持原签到设置和今日标记。
- 地址没有账号时，手动标记接口返回明确错误，避免出现无法落地的标记状态。

## 测试与验证

后端回归测试覆盖：

- 强制不可签到地址可以保存并读取今日手动标记，但公开今日签到状态仍为“不可签到”。
- 手动签到地址的标记在刷新后保持。
- `manual` 和 `disabled` 模式执行检测不会改变 `checkin_mode`。
- `enabled` 模式执行检测仍可更新签到能力。

前端测试覆盖模板中的独立地址标记字段、两种非自动模式下的按钮，以及手动模式不再被归类为“网站不支持签到”。

浏览器验证路径：

1. 打开 `api.e2ez.com` 地址详情。
2. 在“手动签到”下切换今日标记，确认右侧按钮和左侧地址标记同步更新。
3. 刷新页面，确认标记保持。
4. 切换到“不可签到”并手动标记，确认左侧今日签到状态仍为“不可签到”。
5. 对“手动签到”和“不可签到”执行签到状态检测，确认设置不被修改。
6. 对“可以签到”执行检测，确认原有自动更新行为保留。

## 非目标

- 不改变账号余额或登录状态检测逻辑。
- 不移除 `api.e2ez.com` 的强制不可签到规则。
- 不让“不可签到”地址进入全部签到或地址批量签到。
